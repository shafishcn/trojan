# 控制中心发布与升级 SOP

这份 SOP 面向生产环境的控制中心与节点 Agent 升级，目标是把发布过程标准化、可回溯、可回滚。

## 适用范围

- 控制中心版本升级
- Agent 版本升级
- 配置项和运维参数调整
- 涉及 migration 的版本发布

## 发布前检查

发布前至少确认：

- 当前分支已经通过 `go test ./...`
- 当前分支已经通过 `go vet ./...`
- 生产环境最近一次备份可用
- 生产环境 `/readyz`、`/metrics`、`/api/control/alerts/summary` 正常
- 当前没有大面积失败任务和节点失联

## 发布物准备

建议准备以下内容：

- 发布版本号
- 变更说明
- 控制中心二进制
- Agent 二进制
- 对应的回滚版本号

## 正式升级流程

1. 冻结高风险变更
   - 暂停批量用户改动
   - 暂停大规模节点操作

2. 执行备份
   - 调用 `GET /api/control/backup/export`
   - 备份 MySQL
   - 记录备份文件名和时间

3. 测试环境预演
   - 启动同版本 MySQL
   - 导入备份
   - 升级控制中心
   - 检查 migration 是否正常

4. 升级控制中心
   - 停止旧版本服务
   - 替换控制中心二进制
   - 启动新版本服务
   - 检查 `systemctl status`

5. 验证控制中心
   - `GET /readyz`
   - `GET /api/control/runtime/status`
   - `GET /api/control/alerts/summary`
   - 管理员登录
   - 打开控制台首页

6. 分批升级 Agent
   - 先升级 1 台测试节点
   - 验证注册、心跳、拉任务、执行结果
   - 再按区域或批次推进

7. 发布后观察
   - 连续观察 30 到 60 分钟
   - 关注节点在线率、失败任务数、待处理积压

## 验证清单

- 控制中心可以正常登录
- `/readyz` 返回 `200`
- `/metrics` 可被正常抓取
- `/api/control/alerts/summary` 没有异常放大
- 可成功创建一个测试用户
- 可成功给一个测试节点下发 `sync_users`
- Agent 能正常回传任务状态

## 升级窗口建议

- 选择低峰时段
- 避免和大规模节点迁移、线路切换同时进行
- 如果是首个带 migration 的版本，务必先做测试恢复演练

## 升级失败的处理原则

- 优先保业务连续性
- 控制中心异常时，先停止继续升级 Agent
- 确认问题范围后决定回滚还是热修
